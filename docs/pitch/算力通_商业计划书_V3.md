## 第一章 算力——AI 时代的"水电"，一条国家级的确定性赛道

### 1.1 什么是算力？

> **算力（Computing Power）**：简单说就是"计算机处理信息的速度和能力"，衡量单位是 FLOPS（Floating-point Operations Per Second，每秒能执行多少次浮点运算）。
>
> 打个比方：如果把训练一个大模型比作盖一栋摩天大楼，**算力就是施工队的人数**。算力越强，大楼盖得越快。今天最先进的 AI 大模型（如 ChatGPT、DeepSeek），需要成千上万张 GPU 芯片同时跑数个月才能训练出来——这就是"吃算力"。

算力正在成为继**电力、水力、燃气**之后，数字社会第四大公共基础设施。2026 年"十五五"规划首次将全国一体化算力网列入 109 项重大工程，与**水网、新型电网**并列，正式宣告算力进入国家核心基础设施目录。

> **GPU（Graphics Processing Unit，图形处理器）**：原本用于渲染游戏画面的芯片，因为特别擅长做大量并行计算，意外成为 AI 训练的主力芯片。目前全球 AI 训练 90% 以上使用 GPU。主流型号包括 NVIDIA（英伟达）的 A100、H100、H200 等。

### 1.2 市场总览：万亿赛道，智能算力年增 40%+

| 指标 | 2024 年 | 2025 年 | 2026 年（预计） | 同比增速 |
|------|---------|---------|----------------|---------|
| 中国算力市场总规模 | — | **8,351 亿元** | **约 10,500 亿元** | +25%+ |
| 其中：智能算力规模（EFLOPS） | 725.3 | 1,037.3 | 1,460.3 | +41% |
| 全球 GPU 算力租赁市场 | 约 61 亿美元 | 约 74 亿美元 | — | CAGR 21.4% |

> **EFLOPS（Exa FLOPS）**：算力单位。1 EFLOPS = 每秒 100 亿亿次浮点运算。中国的智能算力从 2023 年到 2028 年，五年复合增长率（CAGR，Compound Annual Growth Rate，年均复合增长率）预计达 **46.2%**。

## 第二章 市场空间——用数据算一笔账

### 2.1 TAM / SAM / SOM 自下而上计算

> **TAM（Total Addressable Market，总可触达市场）**：如果整个市场只有你一家，最多能赚多少钱。
> **SAM（Serviceable Addressable Market，可服务市场）**：以你的能力和定位，实际能服务的市场部分。
> **SOM（Serviceable Obtainable Market，可获取市场）**：在竞争中，你实际能拿到的份额。

**我们不从"中国算力市场 8,351 亿"往下拍比例，而是从真实的目标用户数量往上算：**

```
第一步：目标用户基数

中国 AI 开发者总量（2025）               940 万人
  ├── 就职于千人以上大厂（24.3%）        228 万人 → 不直接面向（大厂自建算力）
  ├── 50人以下小团队（20.7%）            195 万人 ✅ 核心目标
  ├── 独立开发者（13.7%）                129 万人 ✅ 核心目标
  └── 其他（学生/科研/中型企业等）         388 万人 → 部分可触达

算力通的核心目标用户池 = 195 万 + 129 万 ≈ 324 万人

第二步：付费意愿筛选

- 其中真正需要 GPU 算力的 AI 开发者（非纯软件/前端）：约 60%
- 实际可用市场规模 = 324 万 × 60% ≈ 194 万人

第三步：平均消费水平

参考 AutoDL（国内最大第三方 GPU 租赁平台）公开价格：
- RTX 4090 按小时 1.6～2.5 元，假设人均月用量 100～200 GPU 小时
- 人均月 GPU 消费中位数：约 300～800 元/月
- 年人均 GPU 消费中位数：约 5,000 元/年

第四步：市场规模计算

TAM（总可触达）= 194 万人 × 5,000 元/年 ≈ 97 亿元/年
SAM（可服务，聚焦创业公司与独立开发者，剔除学生等低付费群体）= 约 50 亿元/年
SOM（可获取，按 3 年后市场份额 0.5%～2%）= 2,500 万元～1 亿元/年
```

> 数据来源：知乎 × 魔搭《2025 年 AI 时代开发者生态白皮书》、AutoDL 平台公开定价

### 3.2 算力消费者的真实困境——"买不起、等不起、配不好"

我们走访和调研了 AI 创业者和个人开发者的典型困境，总结为五大痛点：

**痛点 1：一张 H100 卡售价 20～30 万元，个人开发者根本买不起。**

一张 NVIDIA H100（80GB）芯片市场价约 2.5～3 万美元（约 18～22 万人民币），加上服务器、存储、网络设备、IDC（Internet Data Center，互联网数据中心）托管费用，一台 8 卡 H100 服务器的总拥有成本约 200 万元起步。对于个人开发者和微型创业团队，这是天文数字。

> **IDC（互联网数据中心）**：专门用来放服务器的机房，需要恒温恒湿、稳定供电、高速网络。自建 IDC 是重资产投入，通常以亿为单位。

**痛点 2：大云厂商的后台是为企业运维工程师设计的，不是为 AI 开发者设计的。**

要在阿里云上跑通一个 PyTorch 训练任务，一个新手需要经历：
- 选择合适的 GPU 实例类型（几十种型号）→ 配置 VPC（Virtual Private Cloud，虚拟私有网络）/安全组 → 挂载 NAS（Network Attached Storage，网络存储）→ 安装 CUDA 驱动 → 配置 PyTorch 版本 → 启动 Jupyter Notebook

这个过程对一个熟练运维可能 2 小时，对于一个独立 AI 开发者可能 **2 天**。

> **VPC（虚拟私有网络）**：在云上划出的一块只属于你的网络区域，类似在公共大楼里租了一间独立的办公室。
> **NAS（网络附加存储）**：通过网络连接的文件存储，多台服务器可以同时读写，用来存训练数据和模型。

**痛点 3：价格不透明，同一张卡不同云厂商差 20%～40%。**

没有人帮开发者做比价。阿里云的 A100 可能 18 元/小时，同样配置的华为云可能 15 元/小时——但开发者不知道。闲置时段（如凌晨）价格更可低至正常价 3 折，但普通用户无法实时追踪。

**痛点 4：包月起步，短期需求被强制多付。**

一个开发者可能只需要用 50 小时 GPU 来微调（Fine-tuning，在已训练好的模型基础上用特定数据做小规模再训练）一个开源模型，50 小时的用量不到一周——但大云厂商的 GPU 实例通常按包月起售。用不完的钱，就白花了。

**痛点 5：开发环境每次都要重装。**

每次创建新实例，CUDA（NVIDIA 的 GPU 编程平台，是让 GPU 能跑 AI 计算的底层软件）、PyTorch（Meta 开源的深度学习框架）、Jupyter（交互式代码笔记本，AI 开发者最常用的编程环境）都要重新安装配置。对开发者来说，这是纯粹的时间浪费。

### 3.3 需求爆发式增长——不只是大模型公司在烧钱

AI 算力需求已经不只是几家大模型公司（如 Minimax、智谱、DeepSeek）的事。随着开源大模型（如 LLaMA、Qwen、DeepSeek）的能力逼近闭源模型，越来越多小团队和个人开发者开始基于开源模型做应用层开发：

- **AI Agent（智能体）**：能自主执行任务的 AI 程序，如 AutoGPT——每个 Agent 的每一次"思考"都在消耗 GPU 算力
- **模型微调**（Fine-tuning）：用自己领域的数据调整开源模型，让它更擅长特定任务——比如训练一个法律文书写作模型
- **AI 绘画/视频生成**：Stable Diffusion、Sora 等——每次生成都在消耗 GPU
- **RAG（检索增强生成，Retrieval-Augmented Generation）**：让大模型能"查阅"外部知识库后再回答——企业级应用的主流方案

> **RAG（检索增强生成）**：一种让大模型更聪明的方法。大模型先根据用户问题去外部知识库"查资料"，然后把查到的内容连同问题一起输入模型生成答案。这样既减少了模型"瞎编"的概率，又能用最新数据。RAG 是当前企业采用大模型最主流的技术方案。

这些应用场景的共同特点是：**不需要自研大模型，但离不开 GPU 算力**。这就是算力租赁平台最核心的需求基本盘。

### 3.4 这个市场的结构性机会——为什么大厂看不上市、小厂做不好

当前算力租赁市场的竞争格局：

| 类型 | 代表 | GPU 规模 | 核心用户 | 月均 GPU 消费 | 核心问题 |
|------|------|---------|---------|-------------|---------|
| **云厂商直营** | 阿里云 PAI、华为云 ModelArts | 数万张 | 政企/中大互联网公司 | 5～50 万+ | 后台复杂，不适合个人 |
| **第三方平台** | AutoDL、恒源云 | 1,000～10,000 张 | 学生、独立开发者 | 100～2,000 元 | 以消费级 GPU 为主（4090/3090），企业级 GPU 资源少 |
| **新兴调度平台** | 密瓜智能 | 不持卡 | 大 B 客户 | 10 万+ | 面向企业，非个人开发者 |

**结构性空白在哪里？**

在 **消费级 GPU 平台**（AutoDL）和 **企业级大云厂商**（阿里云）之间，存在一个空白地带：

> 需要 **A100/H100 级别的企业级 GPU**（比 4090 强数倍，适合训练大模型），但只需要按小时用、按秒计费的 **灵活付费方式**（不像云厂商那样包月锁死），且希望 **一键配置好环境**（不像云厂商控制台那么复杂）的 AI 创业者和专业开发者。

这是一群"比学生有钱、比企业灵活"的用户，目前没有一个平台专门为他们设计。

#### 4.5.2 技术栈清单

| 层次 | 技术选型 | 选型理由 |

| **数据库** | PostgreSQL（主库）+ Redis（缓存/会话） | PG 成熟稳定，Redis 用于价格缓存（5 秒刷新）和用户会话 |

#### 4.5.3 核心技术挑战与应对方案

| 技术挑战 | 为什么难 | 应对方案 |
|---------|---------|---------|
| **多云 API 适配** | 每个云厂商的 API 设计差异巨大：实例类型命名、状态码、错误处理、异步操作模式各不相同 | 采用"适配器模式"：每个云厂商一个独立适配器模块，统一转换为内部标准模型。新接入云厂商只需实现一个适配器接口（约 500-800 行代码） |
| **按秒计费精度** | 需要精确到秒级的计费，且要处理实例暂停/恢复、跨月结算、欠费停服不丢数据等边界情况 | 自研计费引擎：基于事件溯源（Event Sourcing）模式，每条计费记录不可变，余额扣减用 Redis 原子操作 + PostgreSQL 异步持久化，确保"不多扣、不少扣、不丢数据" |
| **实时比价的数据一致性** | GPU 价格每几分钟就变（Spot 实例甚至每秒变），如何保证展示给用户的"最优价"真正能买到？ | 分层缓存策略：Redis 缓存现货价格（5 秒刷新），点击下单时实时调 API 二次确认价格（类似机票比价引擎的逻辑）。如果价格已变，展示"价格已更新"提示 |
| **GPU 实例跨云迁移** | 用户想从阿里云迁移到华为云，但训练数据怎么跟着走？（后期功能） | 对象存储中转：训练数据和模型 checkpoint 统一存储在算力通的对象存储中，实例切换时从对象存储拉取，而非云厂商间直接传输（避免跨云带宽费用） |
| **安全隔离** | 用户在 GPU 实例上跑的都是自己代码，如何防止恶意使用（挖矿、攻击、违规内容）？ | GPU 实例运行在用户自己的云账号 VPC 内（非共享主机），算力通只负责创建/释放和计费。加上用量异常检测（如连续 72 小时 100% GPU 使用率 → 人工审核） |

#### 4.5.4 自研 vs 开源依赖

| 组件 | 自研 or 开源 | 决策理由 |
|------|:----------:|---------|
| 多云适配引擎 | **自研** | 核心壁垒。每个适配器约 500-800 行，初始3个云厂商共约 2000 行核心代码 |
| 按秒计费引擎 | **自研** | 核心壁垒。涉及金融级精度，不可用第三方 |
| 比价引擎 | **自研** | 核心体验。推荐逻辑需要根据用户历史行为调优 |
| Web 控制台前端 | **自研** | 核心体验 |
| 镜像管理 | 基于 Docker + Harbor（开源） | 开源成熟方案，不需要重新发明 |
| 用户认证 | 基于 Auth0 / Keycloak（开源） | MVP 阶段用 Auth0 免费额度，后续迁移 |
| 监控告警 | Prometheus + Grafana（开源） | 业界标准 |
| CLI 工具 | **开源（算力通发布）** | 开源策略的一部分——用 CLI 工具吸引开发者进入算力通生态 |

## 第五章 盈利模式——钱从哪里来，账怎么算

### 5.1 三阶段收入结构

                                                                         年费制        5%-10%
综合毛利率: 10%-25%                 综合毛利率: 25%-45%                  综合毛利率: 40%-55%

### 5.2 第一阶段盈利模型详解（0-12 月）

**收入侧——三个来源：**

| 收入来源 | 解释 | 毛利率 |
|---------|------|--------|
| **云厂商返佣** | 通过算力通向阿里云/华为云采购 GPU 实例，云厂商按季度/年度向算力通返佣。类似"携程帮酒店卖房，酒店给携程佣金" | 8%～18%（取决于采购量和代理等级） |
| **平台服务费（差价）** | 在云厂商底价基础上加价 10%～20%，换取一键部署、按秒计费、预置环境等增值服务 | 100%（纯毛利） |
| **增值服务** | 算力券代申领（免费，用于获客）、代运维/模型部署支持（后期收费） | 80%+ |

**实际定价示例——以 A100（80GB）为例：**

| 渠道 | 计费方式 | 单价 | 月费（按 720 小时计） |
|------|---------|------|---------------------|
| 阿里云官方（按量付费） | 按小时 | 18 元/小时 | 12,960 元 |
| 阿里云官方（包月） | 包月 | — | 约 21,000 元 |
| **算力通（标准）** | **按秒** | **20 元/小时** | **约 14,400 元**（但通常用不到 720 小时） |
| **算力通（Spot/闲时）** | **按秒** | **6 元/小时（3 折）** | — |

**对用户的价值：** 如果一个开发者每月实际只用 60 小时 A100：
- 阿里云包月：21,000 元（花了钱但大部分时间没用）
- 算力通标准按秒：1,200 元（只用 60 小时只需付 60 小时的钱）
- **算力通帮她省了 94%。**

**对算力通的利润：**
- 算力通从阿里云拿到底价约 16 元/小时（代理商折扣价）
- 卖 20 元/小时 → 毛利 4 元/小时，毛利率 20%
- 如果是 Spot 闲置实例，拿到底价可能低至 3～5 元/小时，卖 6 元 → 毛利率 20%～50%

### 5.3 单位经济模型（Unit Economics）——假设条件与推导

> **单位经济模型（Unit Economics）**：把一个典型用户从获客到流失的完整生命周期中，所有收入和成本算清楚。用来回答一个核心问题——"每获取一个用户，你到底赚不赚钱？"

**推导过程：**

```
=== 假设条件（保守估算）===

1. 用户获取成本（CAC, Customer Acquisition Cost）：
   - 内容营销（知乎/CSDN/掘金）：免费，靠自然流量
   - GitHub 开源项目引流：免费
   - 付费投放（知乎 CPC ≈ 3-5 元/点击 × 转化率 2%）：约 200 元/付费用户
   - 校园大使计划：每名大使每月补贴 500 元，带来 5 个付费用户 = 100 元/用户
   - 综合 CAC（加权平均）：约 150 元/付费用户

2. 用户月均 GPU 消费（ARPU, Average Revenue Per User）：
   - 参考 AutoDL 用户行为数据（公开可查）：典型用户月均 100-200 GPU 小时
   - 假设算力通用户月均用量：120 GPU 小时/月（取中值）
   - 假设混合 GPU 类型使用：
     ─ 60 小时 × RTX 4090（1.5 元/小时）= 90 元
     ─ 40 小时 × A100（20 元/小时）= 800 元
     ─ 20 小时 × Spot A100（6 元/小时）= 120 元
     ─ 月均总消费 ≈ 1,010 元 → 保守取 600 元（考虑新用户用量偏低）

3. 毛利率：第一阶段取 20%（中间值）

4. 月均用户流失率（Churn Rate）：5% → 用户平均生命周期 = 1/5% = 20 个月
   - SaaS 行业种子阶段平均月流失率 5%-8%，取 5% 为保守值

=== 计算结果 ===

LTV（用户生命周期价值, Lifetime Value）
= ARPU × 毛利率 × 平均生命周期
= 600 × 20% × 20
= 2,400 元

CAC = 150 元

LTV : CAC = 2,400 : 150 = 16 : 1 ✅
（SaaS 健康基准：LTV/CAC ≥ 3:1）

回本周期（Payback Period）= CAC ÷ (ARPU × 毛利率)
= 150 ÷ (600 × 20%)
= 1.25 个月 ✅
```

> **关键结论：** 即使在最保守的假设下（ARPU 600 元、毛利率仅 20%），算力通的单位经济模型依然非常健康——每花 1 元获取用户，能赚回 16 元。

### 5.5 自有 GPU 的经济账——为什么自建比代理赚得多

> **以一个 8 卡 H100 服务器为例，计算自有 GPU 的单位经济模型：**

```
初始投资（CAPEX，Capital Expenditure，资本性支出）：
- 8 张 H100 80GB 卡：8 × 18 万元（渠道价）= 144 万元
- 服务器主机 + 网络 + 存储：约 20 万元
- 合计：约 164 万元/台

运营成本（OPEX，Operating Expenditure，运营性支出）：
- IDC 托管费（含电费、带宽、冷却）：约 1.5 万元/月
- 设备折旧（按 3 年直线折旧）：164/36 = 4.6 万元/月
- 运维人力（分摊）：约 0.5 万元/月
- 合计：约 6.6 万元/月

收入侧（假设 70% 利用率）：
- 8 张 H100 × 24 小时 × 30 天 × 70% = 4,032 GPU 小时/月
- 按 35 元/小时（略低于阿里云官方 43 元的市场价）：
  月收入 = 4,032 × 35 = 141,120 元

单台利润率：
- 月毛利 = 14.1 万 - 6.6 万 = 7.5 万元
- 毛利率 = 7.5/14.1 ≈ 53%

投资回收期（Payback Period）：
- CAPEX ÷ 月净利 = 164 万 ÷ 7.5 万 ≈ 22 个月
- 若部署在西部算力枢纽（享受低电价 + 算力券 30% 补贴）：
  月运营成本可再降 20%，回收期缩短至约 18 个月
```

**这个账说明了什么？** 代理模式的核心意义不在于赚多少利润，而在于**用最小的资本验证 PMF（Product-Market Fit，产品-市场匹配）和积累用户基础**。当付费用户突破 1,500 人（基准方案 M18），平台就具备了启动自有 GPU 部署的"安全垫"——即使自有 GPU 上架初期利用率不高，已有用户基础可以快速消化。

### 5.6 财务敏感性分析

> **为什么需要敏感性分析？** 以上所有财务预测都建立在关键假设之上。如果假设偏了，结果会怎样？敏感性分析帮助投资人理解"这个模型在什么情况下会垮，在什么情况下会飞"。

#### 5.6.1 ARPU 敏感性：如果用户消费只达到预期的一半？

| ARPU 偏离 | 实际ARPU | M12 MRR | M12月净利 | 盈亏平衡时间 | 影响判断 |
|:---------:|---------|--------|----------|:----------:|---------|
| 基准（0%） | 600元 | 30万 | +1.5万 | M10 | — |
| **-30%** | **420元** | **21万** | **-3.5万** | **M16** | ⚠️ 延迟但不致命 |
| -50% | 300元 | 15万 | -8.5万 | M20+ | 🔴 需要额外融资 |
| -70% | 180元 | 9万 | -13.5万 | 2年内不可能 | 🔴 模式不成立 |
| +50% | 900元 | 45万 | +7.5万 | M7 | 🟢 超预期 |

> **关键洞察：** ARPU 下降 30%（从 600 降至 420），盈亏平衡从 M10 推迟到 M16，仍在可接受范围内。但 ARPU 腰斩（≤300元）就需要警惕。**应对预案：** 如果 M6 发现 ARPU 持续低于 400 元，启动两个措施：(1) 提升 ARPU —— 推出"优先调度"付费套餐；(2) 降低 CAC —— 减少付费投放，加大免费渠道投入。

#### 5.6.2 流失率敏感性：如果用户走得比预期快？

| 月流失率 | 平均生命周期 | LTV（ARPU=600，毛利率20%） | LTV/CAC | 判断 |
|:-------:|:----------:|:--------------------------:|:-------:|------|
| 3%（基准偏低） | 33个月 | 3,960元 | 26:1 | 🟢 极健康 |
| **5%（基准）** | **20个月** | **2,400元** | **16:1** | 🟢 健康 |
| 8%（SaaS平均上限） | 12.5个月 | 1,500元 | 10:1 | 🟡 仍健康 |
| 10% | 10个月 | 1,200元 | 8:1 | 🟡 可接受 |
| **15%** | **6.7个月** | **804元** | **5.4:1** | ⚠️ 警戒线 |
| 20% | 5个月 | 600元 | 4:1 | 🔴 需紧急干预 |

> **关键洞察：** 即使在 15% 月流失率（远高于 SaaS 行业平均 5%-8%）的极端情况下，LTV/CAC 仍有 5.4:1，高于 3:1 的健康标准线。**但月流失率 15%+ 意味着产品体验出了大问题，必须从产品层面解决，而非靠财务手段。**

#### 5.6.3 GPU 采购成本敏感性：如果云厂商涨价或返佣政策收紧？

| 云厂商行为 | 对算力通毛利率的影响 | 毛利率变化 | 应对措施 |
|-----------|-------------------|:---------:|---------|
| 返佣从 15% 降至 10% | 代理收入减少 33% | 20%→15% | 提高平台差价 5%（用户端涨价 1 元/小时） |
| 返佣从 15% 降至 5% | 代理收入减少 67% | 20%→11% | 🔴 需紧急接入更多云厂商分散依赖 |
| 阿里云 GPU 涨价 20% | 用户端同步涨价，毛利率不变 | 20%→20% | 对毛利率影响中性，但可能抑制需求 |
| Spot 实例供给减少 | Spot 收入占比下降 | 20%→17% | 引导用户错峰使用，或放宽 Spot 自动竞价策略 |

> **核心风险缓释：** 单云厂商依赖是最大风险。算力通的战略是：**M4 接入阿里云 → M8 接入华为云 → M12 接入腾讯云 → M18 接入 UCloud/青云等二线云。** 当任一云厂商收入占比低于 50% 时，单点风险即可控。

#### 5.6.4 综合压力测试：最坏情况下能撑多久？

```
最坏情景假设（保守方案 × 再恶化 30%）：
─────────────────────────────────────────────
• ARPU：500 × 70% = 350元/月
• 付费用户：300 × 80% = 240人（转化率低于预期）
• MRR：240 × 350 = 84,000元
• 毛利率：18% - 3%（返佣收紧）= 15%
• 月毛利：84,000 × 15% = 12,600元
• 月运营成本：45,000元（已压至最低：2人团队+最小基础设施）
• 月净亏损：-32,400元

资金跑道：200万 ÷ 3.24万/月 ≈ 61个月 ≈ 5年
（现实中还有一次性支出和意外情况，实际约 18-24 个月）

结论：即使在"最坏情况"，种子轮 200 万也足够支撑 18 个月+。
公司有充足时间调整策略、寻找 PMF 或启动过桥融资。
```

### 6.3 从种子轮到 A 轮的触发条件

种子轮投资人最关心的是——**"这笔 200 万花完之后，公司处于什么状态，能不能融到下一轮？"**

算力通的 A 轮触发条件（满足任意两项即可启动）：

| 条件 | 门槛 | 说明 |
|------|------|------|
| 条件 A：收入规模 | MRR ≥ 50 万元 | 连续 3 个月 |
| 条件 B：增长质量 | MRR 月环比增长 ≥ 15% | 连续 6 个月 |
| 条件 C：单位经济模型 | LTV/CAC ≥ 5:1 + 月流失率 ≤ 5% | 证明不是烧钱换增长 |
| 条件 D：自有 GPU 验证 | 至少 4 张自有 GPU 稳定运营 3 个月，毛利率 ≥ 40% | 证明自建模式可复制 |

> **为什么设这些条件？** 2026 年中国一级市场的投资逻辑已经从"烧钱换规模"转变为"小步快跑、里程碑式融资"。投资人现在要看到的是确定性，不是故事。你只有把上一轮的里程碑全部兑现了，他们才会给你下一轮的钱。

### 6.4 数据指标体系——我们衡量什么，就得到什么

> SaaS/平台型公司的核心竞争力之一是将业务量化为一组清晰的数字。算力通从 Day 1 就将建立完整的数据仪表盘，确保每个决策都有数据支持。

#### 6.4.1 北极星指标（North Star Metric）

> **北极星指标 = 月度活跃 GPU 小时数（Monthly Active GPU Hours）**

这个指标同时衡量了：(1) 有多少用户在用（广度），(2) 每个用户用多久（深度），(3) 平台对用户的实际价值（用户愿意花 GPU 小时 = 真正在用）。

### 9.3 差异化护城河分析

算力通不做"更好的 AutoDL"或"便宜的阿里云"，而是开辟一个被双方同时忽视的地带：

#### 护城河 1：结构性空白定位

```
AutoDL 用户     →  需要 A100 但 AutoDL 没有  →  被迫去阿里云  →  发现太贵太复杂  →  放弃/忍受
                                                                    ↑
                                                            算力通 = 这个空白的解决方案
```

这个空白之所以长期存在，是因为：
- **大云厂商不care：** 个人开发者/小团队占云厂商收入不到 5%，不值得为这个群体重新设计产品
- **AutoDL 进不来：** A100/H100 单卡成本是 4090 的 10-20 倍，AutoDL 的轻资产模式（靠量取胜）支撑不了高端 GPU 采购。且其用户群（学生）也付不起 A100 的钱
- **新玩家门槛高：** 需要同时具备"云计算 SDK 集成能力 + AI 框架理解 + 开发者社区运营"，跨界人才稀少

#### 护城河 2：多云调度能力（技术壁垒）

多云适配不是简单调用 API。真正的壁垒在于：
- **错误处理矩阵：** 阿里云有 200+ 种错误码，华为云有自己的错误体系，腾讯云又不同。统一为开发者友好的提示信息需要大量实战积累
- **计费精度：** 跨云按秒计费需要处理各云厂商的不同计费粒度（阿里云按小时出账单但支持按秒查询、华为云按小时出账单按分钟查询），自研计费引擎需要逐一适配
- **实例编排：** 在 3 个云之间做自动故障转移（A 云 GPU 实例创建失败 → 自动尝试 B 云同类实例），需要深刻理解每个平台的限流策略、配额机制

这些不是靠钱能快速买到的——需要工程师花时间踩坑。先发优势在 6-12 个月内有效。

#### 护城河 3：算力券政策壁垒

算力券代申领看似是"政务服务"，但背后涉及：
- 地方政府关系：与各地经信局/大数据局建立联系，获取算力券最新政策
- 申报经验积累：每个地方的申报流程、表格、审批逻辑都不同，做多了就形成"政策知识库"
- 信任资产：当用户在算力通上成功申领到算力券后，迁移到其他平台意味着放弃这条"政策捷径"

#### 护城河 4：合规能力（创始人独有优势）

算力交易涉及 ICP 许可、用户实名、数据安全、增值税合规——任一环节出问题都可能让平台下线。大多数技术创始人要到被约谈时才意识到这些问题。Eric 的律师背景让合规从一开始就是壁垒而非负债。

## 第十二章 风险分析与应对策略

> **为什么需要这一章？** 投资人见惯了"只说好话"的计划书。主动、坦诚地分析风险，并给出经过思考的应对方案，是创始人成熟度的体现。以下风险按可能性和影响程度排序。

### 12.1 风险全景图

```
        影响程度
           ▲
           │
  高影响   │  监管合规风险    云厂商断供      关键人员流失
  中影响   │  价格战          技术复杂度      数据安全
  低影响   │  市场误判        汇率/芯片      竞品抄袭
           │
           └──────────────────────────────────→ 发生概率
              低概率            中概率          高概率
```

### 12.2 各类风险详细分析与应对

---

#### 风险 1：云厂商终止代理协议或收紧返佣政策

| 维度 | 分析 |
|------|------|
| **风险描述** | 阿里云/华为云在某个时间点决定不再与第三方平台合作，或大幅降低返佣比例（如从 15% 降至 3%），导致算力通第一阶段商业模式崩溃 |
| **发生概率** | 🟡 中概率（短期低，长期中） |
| **影响程度** | 🔴 高——如果只依赖单一云厂商，可能致命 |
| **触发信号** | 云厂商推出自己的"个人开发者版 GPU 租赁"产品；云厂商宣布"直销优先"战略；返佣比例连续两次下调 |
| **应对方案** | **(1) 多云分散：** 从 Day 1 就不绑定单一云厂商。M4 阿里云 → M8 华为云 → M12 腾讯云，确保任一云厂商收入占比 < 50%；**(2) 加速自有 GPU 部署：** 如果 M12 时已经验证了 PMF 并有 500+ 付费用户，可以提前启动自有 GPU 采购（即使规模较小，4-8 张卡），降低对代理模式的依赖；**(3) 增加收入多样性：** 增值服务（代运维、模型部署支持）不依赖云厂商返佣 |
| **最坏情况** | 所有云厂商同时收紧政策（概率极低）。此时算力通需立即切换到"纯自有 GPU + 闲置算力撮合"模式，资金跑道决定存活时间 |

---

#### 风险 2：竞品大规模价格战

| 维度 | 分析 |
|------|------|
| **风险描述** | AutoDL 获新一轮融资后，用补贴方式推出 A100 租赁，定价低于算力通；或阿里云大幅降价 GPU 实例 |
| **发生概率** | 🟡 中概率 |
| **影响程度** | 🟡 中——价格不是唯一竞争维度 |
| **应对方案** | **(1) 非价格竞争：** 算力通的核心卖点是"体验"（一键部署、多云比价、算力券代申领），而非单纯"便宜"。价格战打不垮体验优势；**(2) 聚焦高价值用户：** AutoDL 用户群偏价格敏感型，算力通用户更看重"方便+省时间"。价格战对后者影响小；**(3) 差异化定价：** 如果竞品降到成本价以下，算力通可以保持溢价但提供更多增值服务（优先调度、专属技术支持），而不是跟进降价 |
| **历史参考** | 2014-2016 年 O2O 补贴大战：最终胜出的不是补贴最多的，而是体验最好的（美团 vs 饿了么） |

---

#### 风险 3：关键人员风险

| 维度 | 分析 |
|------|------|
| **风险描述** | 技术合伙人迟迟招不到合适人选；或早期核心工程师中途离职；或创始人因健康/家庭原因无法全职 |
| **发生概率** | 🟡 中概率（招人难是初创公司普遍问题），其他低概率 |
| **影响程度** | 🔴 高——种子阶段公司最脆弱的资产是人 |
| **应对方案** | **(1) 招聘漏斗前置：** 在种子轮到账之前就开始运营开源项目和社区内容，让潜在候选人"先看到你"；(2) 期权激励设计：4 年归属期 + 1 年悬崖期（cliff），确保核心团队稳定；(3) 关键人备份：创始人亲自写 MVP，确保即使工程师不到位，产品也能上线；重要代码和云账号权限由至少 2 人持有；(4) 创始人保险：种子轮资金中包含 18 万生活费，确保创始人至少 12 个月无后顾之忧 |
| **最坏情况** | 创始人无法全职 + CTO 未到位。公司需要暂停运营，但法律结构确保投资人权益受保护 |

---

#### 风险 4：监管与合规风险

| 维度 | 分析 |
|------|------|
| **风险描述** | ICP 许可证审批被拒或大幅延迟（>6个月）；算力券代申领被监管部门认定为违规；GPU 出口管制进一步收紧导致云厂商 GPU 供给更加紧张 |
| **发生概率** | 🟡 中概率（ICP 延迟较常见），其他低概率 |
| **影响程度** | 🔴 高——没有 ICP 许可证就不能合法运营 |
| **应对方案** | **(1) ICP 提前启动+双保险：** 公司注册后立即启动 ICP 申请，预计 M8-M10 获批。同期准备"技术内测+免费"模式过渡（不收钱就不需要 ICP）；(2) 算力券合规：代申领定位为"政务信息导流服务"而非"金融中介"，不触碰资金，降低合规风险；**(3) 注册地选择：** 深圳前海/杭州未来科技城对互联网企业审批友好，比在北京/上海注册更容易获批；**(4) 出口管制：** 算力通不直接进口 GPU（代理模式使用云厂商已合规的算力），受出口管制影响间接且有限 |
| **创始人优势** | Eric 的律师背景是应对合规风险的最强武器——他知道要找谁、填什么表、什么话能说什么不能说 |

---

#### 风险 5：技术复杂度被低估

| 维度 | 分析 |
|------|------|
| **风险描述** | 多云适配的工程复杂度远超预期（SDK bug、API 变更、计费精度难以保证）；按秒计费引擎在高并发下出现财务差错；GPU 实例编排（跨云故障转移、状态同步）的边界情况比预期多 |
| **发生概率** | 🟢 高概率（几乎所有技术创业都会遇到） |
| **影响程度** | 🟡 中——技术问题通常可以解决，但会延迟里程碑 |
| **应对方案** | **(1) 渐进式多云接入：** 先做阿里云（最成熟，SDK 最好），跑稳了再做华为云，不要同时开工；(2) 计费引擎独立测试：** 计费引擎从 Day 1 就独立为单独模块，用对账脚本每日自动比对"算力通账单 vs 云厂商账单"，确保分毫不差；**(3) MVP 阶段做减法：** 第一期只支持"创建实例→使用→释放→计费"四个核心操作。GPU 迁移、自动伸缩、跨云故障转移这些高级功能等 M12 以后再考虑；**(4) 踩坑时间预算：** 在开发计划中预留 30% 的"踩坑时间"（4 个月计划实际按 6 个月来预期） |

---

#### 风险 6：市场规模误判——目标用户比预想的少

| 维度 | 分析 |
|------|------|
| **风险描述** | 324 万目标用户池是高估了。可能有大量"AI 开发者"实际上是前端/产品经理/数据分析师，不需要 GPU。真正的目标用户可能只有预估的 30-50% |
| **发生概率** | 🟢 中高概率 |
| **影响程度** | 🟡 中——市场规模仍然足够大 |
| **应对方案** | **(1) 即使只有预估的 30%，市场也不小：** 194 万 × 30% = 58 万人 × 5,000 元/年 = 29 亿元 SAM，对种子轮公司来说足够；(2) 用户基数的自我修正：** M3-M6 的实际注册和付费数据会自然修正市场假设。如果注册转化率远低于预期，立即调整目标用户定义和获客策略；**(3) 向上扩展：** 如果个人开发者市场确实不够大，算力通的"多云比价+统一控制台"同样适用于中型企业的 AI 团队（50-200 人公司），只是获客方式需要从社区转向 BD |

#### 风险 7：数据安全与用户信任

| 维度 | 分析 |
|------|------|
| **风险描述** | 用户担心"算力通会看到我的训练数据和模型"。一旦出现数据泄露事件（即使是用户自身配置错误导致的），平台信任将受到严重打击 |
| **发生概率** | 🟢 低概率（架构设计上天然隔离） |
| **影响程度** | 🔴 极高——信任是算力交易平台的生命线 |
| **应对方案** | **(1) 架构隔离：** GPU 实例运行在用户自有云账号的 VPC 中，算力通技术上无法访问用户数据和代码；(2) 安全白皮书：** 在产品上线时发布《算力通数据安全白皮书》，用技术语言证明"我们看不到你的数据"；**(3) SOC 2 / 等保：** M12 启动等保评测，M24 争取 SOC 2 认证（如果有海外用户）；**(4) 透明日志：** 向用户开放完整的 API 调用日志和实例操作审计日志，建立"你可以验证我们没做坏事"的信任 |

---

#### 风险 8：GPU 芯片出口管制加剧

| 维度 | 分析 |
|------|------|
| **风险描述** | 美国进一步收紧对华 GPU 出口管制（如禁止 H100/H200 对华销售），导致国内高端 GPU 供给极度短缺，云厂商 GPU 实例大面积不可用 |
| **发生概率** | 🟡 中概率（地缘政治不确定） |
| **影响程度** | 🟡 中——对全行业都是冲击，但算力通代理模式反而更灵活 |
| **应对方案** | **(1) 代理模式的优势：** 算力通不持卡，不受芯片禁运直接影响。只要国内还有 GPU 在流通（无论是库存 H100、国产昇腾、还是二线渠道货），算力通就能聚合；(2) 国产 GPU 适配：** M12 开始预研华为昇腾适配器。如果 H100 禁运，国产 GPU 反而迎来爆发——算力通可以快速成为昇腾的最大线上渠道之一；**(3) 东南亚中转：** 如果禁运严格到国内完全拿不到高端卡，可通过新加坡/马来西亚的云厂商节点提供服务（需要额外的数据跨境合规工作） |

## 附录 A：投资人常见问题 FAQ

> 以下问题和回答综合了种子轮投资人最常提出的疑问。建议创始人在路演前将这些问题"内化"——不是背诵答案，而是理解每个问题背后的投资逻辑。

**Q1：你怎么保证云厂商不直接抄你的体验层？**

> **回答：** 大厂不会为了 5% 的用户重新设计产品，就像工商银行不会为了大学生重做手机银行。阿里云的年收入是算力通 SAM 的 200 倍以上——把 PAI 后台改成"个人开发者友好版"需要动整个产品架构，而带来的收入增长对阿里云来说微乎其微。退一步说，如果他们真的抄了，那说明我们选择的市场是真实的——而那时候我们已经有 12-18 个月的先发优势和用户基础。

**Q2：200 万够吗？为什么不是 500 万？**

> **回答：** 种子轮的钱只用来验证 PMF，不是用来买卡的。200 万的资金分配中 43% 是人、20% 是云厂商押金、12% 是推广——全部花在验证"有没有人愿意为多云比价+按秒计费付费"这个核心假设上。如果假设是对的，M10 就能盈亏平衡，不需要更多钱。如果假设是错的，500 万也只是延长了"意识到自己错了"的时间。PMF 验证后，A 轮融 1,500-2,500 万再买资产。

**Q3：你一个人，技术合伙人还没有？**

> **回答：** 开源项目和社区内容已经在做招聘漏斗。更重要的是——MVP 我一个人能写，不需要等 CTO 来划第一根火柴。而且我的策略是先招 3 名工程师，让他们在实际项目中展示能力，M6 前从内部选出或外部引入 CTO。招来的 CTO 远比"找来的"靠谱。

**Q4：AutoDL 已经 10,000 张卡了，你凭什么？**

> **回答：** AutoDL 的主力是 4090 消费级卡，我们做的是 A100/H100 企业级卡——两拨用户基本不重叠。能用 4090 跑通的任务，用户不会多花 10 倍钱买 A100。反过来，需要 A100 的训练任务，4090 显存根本装不下。AutoDL 不做 A100 是有原因的：A100 单卡成本是 4090 的 15 倍，它的学生用户群付不起。我们瞄准的是"比学生有钱、比企业灵活"的中间地带。

**Q5：如果大模型公司自己搞算力平台？**

> **回答：** 大模型公司自己的训练任务已经把所有 GPU 吃满，没有闲置算力可以向外卖。就算有少量闲置，也只能卖 Spot 实例——而算力通本身就是做 Spot 聚合的。大模型公司做算力平台的逻辑相当于"航空公司顺便开旅行社"——可以，但不是主业，也不会投入最好的资源。

**Q6：你为什么要从服装 SaaS 转来做算力？跨度是不是太大？**

> **回答：** 跨度看起来大，但底层能力恰好匹配。服装 SaaS 天花板太低（SOM < 2 亿），算力赛道天花板高 50 倍。而算力赛道需要的三个核心能力——合规（法律）、金融产品设计（算力期货）、技术（多云调度）——恰好是我的三个专业领域。这不是随意跨界，是在自己的能力三角形里找到了最大的商业机会。

**Q7：如果经济下行，AI 投资降温了呢？**

> **回答：** 即使 AI 投资降温，已上线的 AI 应用的推理需求不会消失——它们已经依赖 GPU 为用户提供服务。GPT、DeepSeek、Kimi 的用户每天在用的每一次对话都在消耗算力。这不像区块链是纯投机驱动。退一步说，200 万的规模在寒冬中反而是优势——小船好掉头。

**Q8：用户为什么不直接在阿里云买，要经过你？**

> **回答：** 同一个用户可能有三个场景：今天在阿里云买、明天发现华为云更便宜去华为云——但他不会为了省 3 元/小时每天去三个云平台比价。算力通的价值不是每天帮他省 3 元钱，而是让他不用关心在哪个云上跑、怎么配置环境、怎么申请算力券。这就像你可以在航空公司官网买票，但大多数人还是用携程——因为方便。

**Q9：你的"按秒计费"真的能做到吗？云厂商只支持按小时出账单。**

> **回答：** 技术上可以。云厂商底层 API 支持查询实例的精确启动和释放时间（精确到秒）。我们的计费引擎用事件溯源模式记录每个实例的生命周期，计费用的是我们记录的时间而非云厂商的整点账单。这意味着：用户用 37 分钟，我们收 37 分钟的钱（按秒 × 单价），然后我们向云厂商按整小时结算。差价部分是对冲——用户一年可能少付 20%，但我们通过 Spot 实例的低价买入 + 正常卖出把这部分赚回来。

**Q10：如果 GPU 价格突然腰斩？**

> **回答：** GPU 价格腰斩意味着两个可能：(1) 供给大幅增加（国产 GPU 突破量产）——这对算力通是好事，需求会进一步爆发，虽然利润率可能下降但 GMV 会暴涨；(2) 需求崩塌——这在当前 417% 的需求增速背景下几乎不可能。无论是哪种情况，算力通的代理轻资产模式都不怕价格下跌（我们没有库存贬值风险），反而能帮用户找到更便宜的卡。

**Q11：你的算力券代申领是不是可持续的？政策停了怎么办？**

> **回答：** 算力券是获客手段，不是收入来源。即使算力券政策某天停了，用户在平台上已经有了使用习惯和消费记录，换平台有迁移成本。而实际上，算力券作为"新基建配套政策"正在扩面——2025-2026 年新增了南宁、芜湖等多个城市，趋势是越来越多而非越来越少。

**Q12：你打算什么时候开始收费？免费策略持续多久？**

> **回答：** M2 内测期免费（邀请制，100 人），M3 开始收费。我们不搞长期免费——因为要验证的核心假设就是"有人愿意为方便付费"。如果免费太久，反而无法区分"用户是真的需要产品"还是"因为免费才用"。

**Q13：你的竞争对手是谁？最大的威胁来自哪个方向？**

> **回答：** 短期内没有直接竞争对手——我们占据的是一个结构性空白。最大的威胁不是现有玩家，而是"某个正在车库里做同样事情的人"。所以我们必须在 12 个月内建立三个护城河：用户基础、多云调度技术积累、算力券政策壁垒。

**Q14：你预计什么时候融 A 轮？什么估值？**

> **回答：** 触发 A 轮的条件是 MRR ≥ 50 万（连续 3 个月）或月环比增长 ≥ 15%（连续 6 个月）。预计 M12-M18 之间达到。估值取决于届时 MRR 增速——SaaS 行业种子到 A 的估值通常在投后 5,000 万-1.5 亿之间（对应 MRR 的 10-30 倍 ARR 折价）。

**Q15：如果让你用一个数字说服我投资，是什么？**

> **回答：** 417% vs 128%。这是 2026 Q1 中国 AI 算力需求的增速 vs 供给增速。这是一个 3 倍的剪刀差——而且还在扩大。在这个缺口被填平之前（预计至少到 2028 年国产 GPU 量产后才能开始收窄），算力交易平台有至少 2-3 年的确定性增长窗口。我们不需要发明什么，只需要把已经存在的供给侧和需求侧用更好的产品体验连接起来。

| 术语 | 全称 | 通俗解释 |
|------|------|---------|
| **算力** | Computing Power | 计算机处理信息的速度和能力，单位 FLOPS |

| **FLOPS** | Floating-point Operations Per Second | 每秒浮点运算次数，算力的标准度量单位 |

| **SOM** | Serviceable Obtainable Market | 可获取市场——在竞争中你实际能拿到的份额 |

| **RAG** | Retrieval-Augmented Generation | 检索增强生成，让大模型先"查资料"再回答 |

| **IaC** | Infrastructure as Code | 基础设施即代码，用代码来管理和配置服务器，而不是手动操作 |
| **CAPEX** | Capital Expenditure | 资本性支出，一次性的大额投入（如买服务器） |

| **PLG** | Product-Led Growth | 产品驱动增长——靠产品本身（而非销售团队）获取和留住用户 |
